# 0001. Выбрать Blue-Green деплой для ML-сервиса

Date: 2026-05-17

## Status

Accepted

## Context

В сервисе мало обработки ошибок, поэтому новую версию модели нельзя сразу
ставить на весь трафик.

Нужно проверить:

- старая и новая версии отвечают на `/health`
- новая версия отвечает на `/predict`
- есть быстрый возврат на старую версию
- CI/CD показывает, что тесты и model gate прошли

## Decision

Выбран Blue-Green деплой.

- Blue - текущая stable-версия `v1.0.0`
- Green - новая версия `v1.1.0`
- nginx отправляет трафик на активный цвет
- rollback - это переключение nginx обратно на Blue

## Alternatives

Canary тоже подходит для ML-сервисов, но там нужно честно делить трафик
например 90/10, потом 50/50 и только потом 100%. Для этой домашки это сложнее
доказать, потому что нет реального потока пользователей.

Shadow еще безопаснее: новая модель получает копию запросов, но ее ответ не
видит пользователь. Минус - это больше дизайн, чем норм
демонстрация.

## Consequences

Плюсы:

- green можно проверить до переключения трафика
- rollback короткий и понятный
- легко сделать скрины `/health`, `/predict`, switch и rollback

Минусы:

- одновременно работают две версии сервиса
- в настоящем проде надо отдельно проверять совместимость данных и схем
- Blue-Green сам по себе не показывает постепенную реакцию пользователей, как Canary или A/B

## Rollback rule

Возвращаемся на Blue, если:

- `/health` не отвечает `200`
- `/predict` падает
- p95 latency выше 500 ms
- error rate выше 1%
- offline model gate не прошел
